## 5. Жёсткие ограничения (нарушение недопустимо)

1. **`organization_id` обязателен в каждой таблице** — multi-tenancy в каждом WHERE. Нет исключений. → ADR-012.
2. **Не вводить новые расширения PostgreSQL** без нового ADR.
3. **Не изменять принятый ADR** — если решение устарело, создать новый ADR с пометкой "Заменяет ADR-NNN".
4. **`backend/database/schema.sql` — DDL-эталон.** Любое изменение схемы сначала вносится сюда, затем синхронизируется в `schema.prisma` и генерируется миграция.
5. **Prisma-схема должна проходить `prisma validate`** после каждого изменения. Не коммитить невалидную схему.
6. **Патчи нумеруются последовательно.** Следующий патч — P-33 (текущий эталон — P-01..P-32). Номер не повторять.
7. **Не трогать** `apps/`, `packages/shared/`, `infra/docker/docker-compose.yml` — вне зоны ответственности.

## 7. Правила работы с миграциями

```
Создавать миграцию вручную:
  packages/database/prisma/migrations/NNNN_<описание>/migration.sql

Формат имени: 4 цифры порядкового номера + snake_case описание
Пример:       0002_add_work_volume_partitioning

После создания файла миграции:
  1. Обновить schema.prisma (синхронно с migration.sql)
  2. Обновить backend/database/schema.sql (DDL-эталон)
  3. Запустить: cd packages/database && pnpm prisma validate
  4. Добавить в CLAUDE.md §2 запись о новом патче P-NN (если схема менялась)
```

## 8. Правила работы с Materialized Views (Этап 5)

- MV создаются в отдельных `.sql`-файлах: `backend/database/mv_<имя>.sql`
- Каждый MV — отдельный файл с комментарием: назначение, частота рефреша, зависимые таблицы.
- Рефреш MV — только через BullMQ worker (не cron напрямую). → ADR-005.
- Индексы на MV — в том же файле, что и CREATE MATERIALIZED VIEW.
- Данные аналитики (формулы pct_ready, WMA, прогнозы) — читать `docs/algorithm_v1_3.md` перед созданием MV.

---

## 9. Правила работы с pg_partman (Этап 12)

- Партиционирование применяется только к таблицам из ADR-010 (читать перед началом).
- Конфиг pg_partman выносится в `backend/database/partitioning/setup.sql`.
- После настройки партиционирования — добавить патч P-NN в `schema.sql`.
- Политика retention (удаление старых партиций) — согласовать с архитектором до реализации, не придумывать самостоятельно.

---

## 10. Правила оптимизации запросов (Этап 11)

- Перед добавлением индекса — получить `EXPLAIN (ANALYZE, BUFFERS)` на реальных данных (seed или production-like).
- Индексы документировать в `backend/database/schema.sql` с комментарием: какой запрос покрывает.
- Partial indexes — предпочтительны над full indexes для фильтрации по `organization_id` + статус.
- Не добавлять индексы "на всякий случай" — только на подтверждённые медленные запросы.
- PgBouncer: режим `transaction` по умолчанию. Изменение режима — только через новый ADR.
