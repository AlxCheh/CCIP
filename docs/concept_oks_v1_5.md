# Глоссарий и договорённости по терминологии

| **Термин** | **Определение** |
| --- | --- |
| Наименование вида работ | Единица структуры объёмов строительных работ. Заменяет формулировки «раздел», «позиция», «пункт работ». Используется на всех уровнях: от укрупнённого раздела до отдельной строки ведомости. |
| ОКС | Объект капитального строительства — любое здание, сооружение или комплекс, к которому применима данная концепция. |
| 0-отчёт | Зафиксированный факт выполнения работ на дату подключения системы. Является точкой отсчёта, а не первым периодом ввода данных. |
| Период | Единица времени для регулярного обновления данных, задаётся диапазоном дат. Открывается и закрывается стройконтролем или администратором. Закрытый период недоступен для редактирования. Изменение закрытого периода допускается только администратором с фиксацией основания и даты. |
| Плановая пауза | Период, переведённый стройконтролем или администратором в статус паузы в любой момент пока период открыт — как при открытии, так и после. Означает отсутствие производства работ (праздники, ожидание поставки, технологический перерыв и др.). Пауза может быть отменена с обязательным примечанием. Исключается из расчёта темпа. Основание обязательно; при выборе «Иное» — примечание обязательно. |
| Плановый темп | Остаток планового объёма ÷ количество оставшихся периодов до плановой даты завершения. Пересчитывается автоматически при сдвиге сроков. |
| Фактический объём | Суммарный объём выполненных работ по виду работ от начала строительства по состоянию на отчётную дату. |
| % готовности | Отношение фактического объёма к плановому объёму по виду работ, разделу или объекту в целом. На уровне объекта — взвешенный показатель. Не превышает 100% на уровне вида работ даже при фактическом перевыполнении. |
| Весовой коэффициент | Доля договорной стоимости вида работ в общей договорной стоимости объекта. Рассчитывается от РДЦ (при отсутствии — от ССР). Привязан к договорной стоимости, не пересчитывается при изменении факта. |
| Заявленный объём | Объём выполненных работ, указанный ГП в шаблоне-сводке. До верификации стройконтролем не является подтверждённым фактом. |
| Верифицированный факт | Объём, подтверждённый строительным контролем Заказчика по результатам осмотра объекта. Является единственным основанием для ввода данных в систему. |
| Расхождение | Разница между заявленным объёмом ГП и верифицированным фактом стройконтроля по одному виду работ за один период. |
| РДЦ (Расчёт договорных цен) | Источник договорной стоимости по каждому виду работ. Основание для расчёта весовых коэффициентов. При отсутствии РДЦ временно используется ССР. |
| Флаг «Спорное» | Явное действие стройконтроля в карточке верификации: снятие отметки «Работы доступны для осмотра и замера». Переводит позицию в Тип 2 (спорное расхождение) и статус «Оспорено». Основание обязательно. |
| Критический путь | MAX(forecast_end) по видам работ с weight_coef ≥ weight_threshold или из списка critical_works[]. Отражает реальную дату готовности объекта. |
| Версия BoQ | Зафиксированное состояние ведомости объёмов и РДЦ на момент изменения. Любое изменение L2 создаёт новую версию. Версии хранятся в журнале изменений BoQ. |
| decay_factor | Коэффициент затухания для взвешенного среднего темпа. Определяет, насколько быстро снижается вес более ранних периодов. Дефолт 0.9. Задаётся на Уровне 0. |

# 1. Концептуальный каркас

Концепция построена на универсальном принципе: любой объект капитального строительства управляется по единой четырёхуровневой модели.

| **Уровень** | **Название** | **Содержание** |
| --- | --- | --- |
| 0 | Конфигурация системы | Настраиваемые параметры (полный перечень):<br>— Длина периода<br>— N_flag_threshold (дефолт 3) и M_flag_window (дефолт 5) — флаг системных расхождений<br>— weight_threshold (дефолт 0.10) — порог веса для критического пути<br>— forecast_gap_alert (дефолт 2 периода) — порог флага разрыва прогнозов<br>— tolerance_threshold (дефолт 5%) — допустимое превышение плана<br>— overrun_warning_limit (дефолт 20%) — порог жёсткого стопа при перевыполнении<br>— avg_pace_periods (дефолт 5) — ширина окна темпа<br>— decay_factor (дефолт 0.9) — затухание весов в расчёте темпа<br>— zero_report_alert_days (дефолт 5) — таймер утверждения 0-отчёта<br>— baseline_correction_threshold (дефолт 5%) — порог для квалификации правки 0-отчёта<br>— work.tolerance_pct — допустимая погрешность по виду работ (задаётся на уровне позиции)<br>— Список critical_works[] — влияет на обязательность фото, критический путь и перекрёстную верификацию 0-отчёта<br><br>Изменяются только администратором. |
| 1 | Паспорт объекта | Неизменяемые данные: идентификация, технические параметры, участники проекта. Заполняется один раз. |
| 2 | Проектная документация | Плановая точка отсчёта: ССР, РДЦ, календарный план, ведомость объёмов работ. Версионируется при любом изменении. Обновляется администратором с утверждением директора. |
| 3 | Текущий статус | Живые данные: 0-отчёт, периодическое внесение факта, расчёт % готовности, отклонения, прогноз. |

# 2. Структура исходных данных

## 2.1 Паспорт объекта (Уровень 1)

Анкета объекта, которая служит контекстом для всей последующей аналитики. Для складского центра класса А включает:

- Наименование и функциональное назначение объекта
- Класс объекта (A, B, C)
- Общая площадь (м²), этажность, конструктивная схема
- Заказчик / застройщик, генеральный подрядчик, проектная организация
- Реквизиты разрешения на строительство

Технические параметры объекта не изменяются. Данные об участниках проекта актуализируются администратором с фиксацией даты и основания изменения.

## 2.2 Проектная документация (Уровень 2)

| **Документ** | **Роль в системе** |
| --- | --- |
| Сводный сметный расчёт (ССР) | Справочная база при отсутствии РДЦ. |
| РДЦ (Расчёт договорных цен) | Основной источник договорной стоимости по каждому виду работ. |
| Календарный план | Определяет плановые сроки и вехи по этапам строительства. |
| Ведомость объёмов работ (BoQ) | Формирует структуру наименований видов работ с единицами измерения и плановыми объёмами. Группируется до ≤50 позиций при инициализации системы. |

## 2.3 0-отчёт (Уровень 3, однократно)

Поскольку система подключается к уже строящемуся объекту, первым действием является фиксация фактического объёма по всем видам работ на дату подключения. 0-отчёт — фундамент системы: ошибка в стартовом факте искажает все последующие расчёты.

### Иерархия источников стартового факта

| **Приоритет** | **Источник** | **Условие применения** |
| --- | --- | --- |
| 1 (приоритет) | Натурный замер стройконтролем | Всегда, когда физически возможен |
| 2 | Исполнительная документация ГП | При невозможности натурного замера |
| 3 | Акты выполненных работ (КС-2) | При отсутствии исполнительной документации. Обязательная пометка об источнике в карточке. |

### Допустимая погрешность

Задаётся на уровне вида работ в ведомости администратором при настройке (параметр `work.tolerance_pct`). Если разница между натурным замером и документацией ГП превышает порог — позиция получает флаг «требует дополнительной верификации». Флаг не блокирует 0-отчёт, но требует примечания стройконтроля.

### Перекрёстная верификация

| **Категория позиции** | **Требуемая доказательная база** |
| --- | --- |
| weight_coef ≥ weight_threshold (дефолт 0.10) ИЛИ позиция из critical_works[] | Три документа обязательны:<br>1. Натурный замер с фото<br>2. Исполнительная схема с подписью ГП<br>3. КС-2 как контрольная точка<br><br>Директор утверждает только при наличии всех трёх. |
| Остальные позиции (малый вес) | Достаточно одного источника с примечанием об источнике. |

### Таймер утверждения

Первый период не может быть открыт без утверждённого 0-отчёта — жёсткая блокировка. Если 0-отчёт не утверждён за `zero_report_alert_days` (дефолт 5 календарных дней) — администратор получает автоматическое уведомление. Уведомление повторяется циклически каждые `zero_report_alert_days` дней до утверждения.

### Исправление 0-отчёта после запуска системы

| **Случай** | **Признак** | **Процедура** |
| --- | --- | --- |
| A — Техническая ошибка | Правка ≤ baseline_correction_threshold (дефолт 5% от plan_volume) | Администратор исправляет с аудит-логом. Последующие периоды пересчитываются автоматически. |
| B — Системная ошибка | Правка > baseline_correction_threshold | Решение директора проекта. Основание обязательно. Все затронутые закрытые периоды маркируются «пересчитано [дата], основание: [причина]». История не удаляется. |

# 3. Участники процесса оцифровки

| **Участник** | **Роль и ответственность** |
| --- | --- |
| Генеральный подрядчик (ГП) | Источник первичных данных. Предоставляет заполненный шаблон-сводку выполненных объёмов за период. В системе не работает. |
| Строительный контроль Заказчика | Единственный оператор системы. Верифицирует данные ГП через осмотр объекта и фотофиксацию. Вносит верифицированный факт в систему. Классифицирует расхождения. Отмечает плановые паузы. Инициирует UpdateBaseline и изменения L2. |
| Директор проекта (Заказчик) | Получатель аналитики. Видит дашборд, отчёты, флаги. Утверждает 0-отчёт, UpdateBaseline, изменения L2, решения по активным спорам (Сценарий B дедлока). |
| Администратор системы | Назначается Заказчиком. Настраивает конфигурацию (Уровень 0). Утверждает UpdateBaseline и изменения L2 совместно с директором. Изменяет закрытые периоды с фиксацией основания. |

## 3.1 Процедура смены Генерального подрядчика

Смена ГП выполняется только между периодами — аналогично правилу UpdateBaseline. Если период открыт — смена заблокирована.

**Условия для смены:** текущий период закрыт, все расхождения урегулированы (0 позиций в статусе «Оспорено»).

**После смены:** новый ГП получает уведомления Типа 1, шаблон следующего периода формируется на его имя, дата перехода ответственности фиксируется в аудит-логе администратором.

История, журнал расхождений и закрытые периоды остаются без изменений — не переназначаются.

# 4. Формат сводки от Генерального подрядчика

## 4.1 Принцип шаблона-выгрузки

Шаблон формируется автоматически при открытии периода стройконтролем. Генеральный подрядчик вносит только два поля: объём за период и примечание.

## 4.2 Трёхэтапный цикл верификации

| **Этап** | **Участник** | **Действие** | **Срок** |
| --- | --- | --- | --- |
| 4а | ГП → Стройконтроль | Передача заполненного шаблона-сводки | До установленного дня периода |
| 4б | Стройконтроль | Осмотр объекта, сверка данных ГП с натурой, фотофиксация | В течение 2 календарных дней |
| 4в | Стройконтроль → Система | Ввод верифицированного объёма, закрытие периода | До конца отчётного периода |

# 5. Механика оцифровки факта

## 5.1 Шесть шагов процесса

| **Шаг** | **Действие** | **Описание** |
| --- | --- | --- |
| 1 | Структура видов работ | Формируется из ведомости объёмов РД. Группируется до ≤50 позиций администратором при инициализации. |
| 2 | Плановые объёмы и ед. изм. | Для каждого вида работ: единица измерения + плановый объём из РД + допустимая погрешность (work.tolerance_pct). |
| 3 | 0-отчёт | Фиксация фактических объёмов на дату подключения по методике §2.3. |
| 4 | Периодическое внесение факта | Трёхэтапный цикл верификации (см. §4.2). |
| 5 | Расчёт % готовности | MIN(факт/план × 100, 100) — по виду работ, разделу, объекту. |
| 6 | Аналитика и отклонения | Факт vs план, два прогноза, флаги расхождений и разрыва прогнозов. |

## 5.2 Формальная классификация расхождений

Классификация выполняется через два бинарных вопроса, исключающих серую зону:

```
Вопрос 1: Работы физически доступны для осмотра и замера?
Вопрос 2: Замер можно выполнить без разрушения конструкции?

Доступны = ДА  И  Замер без разрушения = ДА  →  Тип 1 (техническое)
Доступны = НЕТ ИЛИ Замер без разрушения = НЕТ →  Тип 2 (спорное)
```

| **Тип** | **Формальные признаки** |
| --- | --- |
| Тип 1 — Техническое (все три условия) | 1. Работы видны и доступны на момент выезда<br>2. Объём измерим без разрушения<br>3. Расхождение — ошибка замера, округление, опережающий учёт |
| Тип 2 — Спорное (достаточно одного) | 1. Работы скрыты (залиты, засыпаны, закрыты конструкцией)<br>2. Зона физически недоступна на дату верификации<br>3. Замер требует разрушения или демонтажа<br>4. Документация ГП отсутствует, неполна или противоречива |

### Закрытые пограничные случаи

| **Случай** | **Правило** |
| --- | --- |
| Частичная скрытость | Если хотя бы часть объёма недоступна — весь объём → Тип 2. На усмотрение стройконтроля: может принять видимую часть как Тип 1 с обязательным примечанием. |
| Несвоевременный выезд стройконтроля | Зона закрыта ДО планового срока выезда → Тип 2 по вине ГП (стандартная процедура). Зона закрыта ПОСЛЕ планового срока выезда → Тип 2 с пометкой «верификация не выполнена в срок». Инцидент фиксируется в журнале с именем инженера. |
| delta = 0, документов нет | Статус «Подтверждено» на основании осмотра. Исполнительная документация — условие подписания КС-2, не верификации в системе ОКС. |

# 6. Регламент урегулирования расхождений

## 6.1 Типология расхождений

| **Тип расхождения** | **Признак** | **Действие** |
| --- | --- | --- |
| Тип 1 — Техническое | Оба условия §5.2 выполнены. Работы доступны, замер возможен. | Стройконтроль вносит свою цифру. Авто-уведомление ГП. Период не блокируется. |
| Тип 2 — Спорное | Стройконтроль снимает отметку «Работы доступны» в карточке верификации. Основание обязательно. Действие логируется с датой и именем инженера. | Статус «Оспорено». Запускается процедура §6.2. Период не закрывается до урегулирования. |
| Тип 3 — Системное | ≥ N_flag_threshold расхождений Типа 2 за последние M_flag_window периодов (скользящее окно). Тип 1 не учитывается. | Автофлаг на дашборде директора: «N спорных за M периодов» + накопленная дельта. Снимается вручную директором. |

## 6.2 Алгоритм урегулирования спорного расхождения (Тип 2)

1. Стройконтроль фиксирует позицию со статусом «Оспорено». Период по данной позиции не закрывается до урегулирования.
2. ГП получает запрос с шаблонным чеклистом требуемых документов: исполнительная схема с подписью, журнал работ, акт на скрытые работы, фото до закрытия.
3. ГП предоставляет подтверждение в течение 2 календарных дней.
4. При отсутствии ответа применяется SLA-процедура §6.6 (Deadlock).
5. Позиция переходит в статус «Подтверждено», запись в журнале расхождений закрывается.

## 6.3 Журнал расхождений

| **Поле** | **Содержание** |
| --- | --- |
| Вид работ | Ссылка на позицию ведомости объёмов |
| Период | Диапазон дат периода |
| Заявлено ГП | Объём из шаблона-сводки |
| Подтверждено | Верифицированный объём стройконтроля |
| Дельта | Абсолютная разница и % отклонения |
| Тип | Техническое / Спорное / Системное |
| Статус | Оспорено / Подтверждено / Закрыто / Спор не урегулирован |
| Основание | Примечание стройконтроля, ссылка на фото или документ |
| Дата закрытия | Дата урегулирования расхождения |
| Закрыл (инженер) | Имя инженера стройконтроля, закрывшего позицию |

## 6.4 Флаг системного расхождения

Система автоматически выявляет паттерн через скользящее окно шириной `M_flag_window` периодов. В счёт идут только расхождения Типа 2 — технические расхождения (Тип 1) не учитываются для целей флагирования.

На дашборде директора проекта появляется флаг с указанием вида работ, количества спорных расхождений в окне и накопленной дельты. Флаг снимается вручную директором с фиксацией даты и основания.

## 6.5 Стандарт верификации

### Критерии «факт подтверждён» — все три условия обязательны

1. Объём измерен или визуально верифицирован стройконтролем лично.
2. Доказательная база собрана согласно категории работ (таблица ниже).
3. Данные внесены в систему в течение 2 календарных дней с момента осмотра.

### Минимальный набор доказательств

| **Доказательство** | **Категория 1: открытые работы** | **Категория 2: скрытые работы (Тип 2)** |
| --- | --- | --- |
| Натурный замер | Обязателен | — (невозможен) |
| Фото с привязкой к карточке верификации | Обязательно для critical_works[], опционально для остальных | Обязательно (до закрытия конструкцией) |
| Исполнительная схема с подписью ГП | Опционально | Обязательно |
| Акт на скрытые работы | — | Обязательно |
| КС-2 | Опционально | Рекомендуется |
| BIM-ссылка на элемент модели | Дополнительно (если BIM подключён) | Дополнительно |
| Геотег фото | Опционально | Опционально |

### Стандарт качества фотофиксации

| **Требование** | **Описание** |
| --- | --- |
| Ориентир привязки | На фото должна быть видима ось, отметка или номер секции |
| Масштаб | Рядом с объектом — рулетка или известный размерный ориентир |

| Геотег | Опционально — если устройство поддерживает, координаты прикладываются автоматически |

## 6.6 SLA deadlock периода

### Сценарий A — молчание ГП

| **День** | **Действие** |
| --- | --- |
| День 0 | Тип 2 выставлен. ГП получил запрос с чеклистом документов. |
| День 2 | Дедлайн ответа ГП. |
| День 3 | Автоуведомление директору проекта: «ГП не ответил по позиции X». |
| День 5 | Принудительное закрытие стройконтролем: вносит верифицированный объём по осмотру. Статус: «Подтверждено (принудительно)». Журнал: «Закрыто без ответа ГП, день 5». Период разблокируется. |

### Сценарий B — активный спор

| **День** | **Действие** |
| --- | --- |
| День 0 | ГП предоставил документацию, стройконтроль не принял. |
| День 2 | Стройконтроль фиксирует основание отказа в карточке (обязательно). |
| День 3 | Автоэскалация директору: «Активный спор по позиции X, обе стороны предоставили документацию». |
| День 7 | Директор принимает решение: утвердить объём стройконтроля / утвердить объём ГП / назначить независимую экспертизу. |
| День 14 | Если решения нет — система применяет объём стройконтроля как консервативную оценку. Статус: «Спор не урегулирован». Период закрывается. |

> Все сроки — календарные дни. Ответственность разграничена датами: закрытие зоны ДО планового выезда стройконтроля — Тип 2 по вине ГП; закрытие ПОСЛЕ — Тип 2 с пометкой о несвоевременном выезде стройконтроля.

# 7. Ключевые принципы интерфейса

## 7.1 Навигация: клик по виду работ → интерактивная карточка ввода

| **Элемент карточки** | **Содержание** |
| --- | --- |
| Шапка | Раздел, наименование вида работ, период, единица измерения |
| Блок контекста | Плановый объём / фактический объём до периода / остаток / плановый темп — только чтение |
| Поле ввода факта | Числовое поле «Объём выполнен за период» + опциональное примечание |
| Флаг «Спорное» | Чекбокс «Работы доступны для осмотра и замера». Если снят — позиция переходит в Тип 2. Обязательное поле «Основание» при снятии. Фото обязательно (доказательная база по §6.5 Категория 2). |
| Прогресс-бар | Визуализация % готовности. Не превышает 100% даже при факте > план. |
| История | Лента предыдущих периодов с объёмами, версией BoQ и признаком плановой паузы |

## 7.2 Логика блокировки полей

Завершённые виды работ (100%) автоматически блокируются. Поле «Фактический объём до периода» всегда подтягивается из предыдущего периода — недоступно для ручного редактирования.

При статусе «Оспорено» поле ввода остаётся доступным. При превышении `overrun_warning_limit` — ввод заблокирован до UpdateBaseline или акцепта администратора.

## 7.3 Чеклист периода

Перед закрытием периода система отображает сводный чеклист: внесено / ожидают ввода / в статусе «Оспорено» / плановые паузы. Наличие позиций в статусе «Оспорено» блокирует закрытие периода.

## 7.4 Периодичность обновления

Длина периода задаётся на Уровне 0 администратором. Для складского центра класса А рекомендуемая длина — одна неделя.

## 7.5 Диалог разрешения дубликатов при импорте ВОР

При обнаружении двух одинаковых наименований видов работ при импорте ведомости — система блокирует создание позиций и показывает обе строки. Пользователю предлагается три действия:

1. **Переименовать одну из позиций** — обе остаются в BoQ с разными наименованиями.
2. **Это одна позиция — объединить** — объёмы суммируются, создаётся одна позиция.
3. **Отменить импорт** — вернуться к редактированию BoQ.

Без выбора одного из трёх вариантов сохранение заблокировано.

## 7.6 Офлайн-режим и синхронизация

Стройконтроль работает локально на мобильном устройстве — вводит объёмы, делает фото, выставляет флаги. Все действия записываются в локальную очередь с временными метками. Закрытие периода — только онлайн.

| **Сценарий синхронизации** | **Поведение системы** |
| --- | --- |
| Период открыт | Данные применяются штатно. Временная метка — дата офлайн-ввода. |
| Период закрыт пока устройство было офлайн | Синхронизация блокируется. Администратор получает уведомление и принимает решение: принять данные (открыть период) или отклонить. |
| Конфликт данных (другой пользователь внёс данные по той же позиции) | Система показывает обе версии с именем, датой и значением. Стройконтроль выбирает актуальную версию с обязательным примечанием. Last-write-wins не применяется. |

# 8. Расчёт % готовности объекта

## 8.1 Взвешенный показатель

```
% готовности объекта = Σ (MIN(% вида работ, 100%) × вес вида работ)
```

## 8.2 Статусы видов работ

| **Статус** | **Условие** | **Действие системы** |
| --- | --- | --- |
| Не начато | % = 0 | Поля ввода активны, нулевой прогресс-бар |
| В работе | 0% < % < 100 | Поля ввода активны, отображается прогресс-бар |
| Оспорено | Флаг «Спорное» выставлен стройконтролем | Позиция блокирует закрытие периода. Дашборд отображает % по последнему закрытому периоду с пометкой «данные актуальны на [дата]». |
| Подтверждено | Спорное расхождение урегулировано | Позиция учитывается в % готовности. Запись в журнале закрывается. |
| Готово | % = 100 | Поля ввода заблокированы, раздел сворачивается |
| Исключена из scope | Структурное изменение L2: позиция с ненулевым фактом удалена из BoQ | Факт сохраняется в истории. Позиция выведена из расчёта % и весов. |

## 8.3 Обработка превышения планового объёма (>100%)

| **Диапазон превышения** | **Поведение системы** |
| --- | --- |
| Факт ≤ план × (1 + tolerance_threshold) дефолт: до 5% | Предупреждение в карточке. Примечание опционально. Ввод разрешён. |
| Факт в диапазоне от tolerance_threshold до overrun_warning_limit дефолт: 5–20% | Примечание стройконтроля обязательно. Флаг «превышение плана» на дашборде директора. Рекомендуется запустить UpdateBaseline. |
| Факт > план × (1 + overrun_warning_limit) дефолт: >20% | Жёсткий стоп. Ввод заблокирован. Требуется UpdateBaseline ИЛИ явный акцепт администратора с основанием. |

> % готовности никогда не превышает 100% на уровне вида работ: `pct_ready = MIN(факт / план × 100, 100)`. Веса не пересчитываются при перевыполнении — привязаны к договорной стоимости. Фактический объём сохраняется полностью.

# 9. Аналитические срезы

| **Срез** | **Основные показатели** | **Применение** |
| --- | --- | --- |
| Финансы | Освоение бюджета, перерасход/экономия по разделам, прогноз итоговой стоимости | Контроль инвестора / заказчика |
| Сроки | % готовности vs план-график, отставание в периодах, два прогноза завершения: взвешенный и по критическому пути, флаг разрыва | Контроль отставания, основание для эскалации |
| Качество | Открытые замечания, срок устранения, динамика закрытия | Технический надзор |
| Ресурсы | Численность рабочих, техника, поставки материалов | Логистика и планирование |
| Расхождения | Флаги системных расхождений (≥N Типа 2 за M периодов), накопленная дельта, активные споры | Контроль достоверности данных, эскалация |

## 9.1 Два прогноза на дашборде директора

| **Показатель** | **Формула** | **Назначение** |
| --- | --- | --- |
| Взвешенный прогноз | weighted_forecast(all_works) | Отражает общий темп освоения бюджета |
| Прогноз по критическому пути | MAX(forecast_end) по видам работ с weight_coef ≥ weight_threshold ИЛИ из critical_works[] | Реальная дата готовности объекта |

**Флаг разрыва** — отдельный автоматический флаг на дашборде. Появляется когда разрыв между прогнозами ≥ `forecast_gap_alert` периодов. Содержит наименование вида работ-узкого места. Снимается автоматически при сокращении разрыва ниже порога.

## 9.2 Взвешенный темп с затуханием

Для расчёта прогноза используется взвешенное скользящее среднее — более поздние периоды имеют больший вес:

```
pace_weighted = Σ(volume[i] × decay_factor^(n-i)) / Σ(decay_factor^(n-i))

Параметры (Уровень 0):
  avg_pace_periods = 5     (ширина окна)
  decay_factor     = 0.9   (затухание, корректируется по фактическим данным)

Пример при decay=0.9, окно 5 периодов:
  P(n): вес 1.0 │ P(n-1): 0.9 │ P(n-2): 0.81 │ P(n-3): 0.73 │ P(n-4): 0.66
```

### Обработка нулевых периодов

| **Тип нуля** | **Признак** | **Обработка в расчёте темпа** |
| --- | --- | --- |
| А — Плановая пауза | Стройконтроль или администратор перевёл период в статус паузы (в любой момент пока период открыт) с обязательным основанием. Отмена паузы возможна с примечанием. | Исключается из окна расчёта темпа |
| Б — Внеплановый простой | Нулевой объём без отметки о паузе | Включается со стандартным весом. Предупреждение директору. |
| В — Позиция не начата | Периоды до старта работ по виду | Исключаются из окна расчёта темпа |

### Обработка выбросов

Выброс: `volume[i] > pace_weighted × spike_threshold` (дефолт 2.0, параметр Уровня 0).

При обнаружении: флаг в карточке, стройконтроль объясняет в течение 1 периода. Если «плановая концентрация» — вес понижается до 0.5. Если «ошибка ввода» — период исключается. Без реакции — автоматически понижается вес до 0.5.

# 12. Версионирование проектной базы (L2)

## 12.1 Принципы версионирования

Любое изменение L2 создаёт новую версию BoQ — независимо от типа и масштаба изменения. Версии нумеруются инкрементально: v1.0, v1.1, v1.2 и т.д.

| **Атрибут версии** | **Содержание** |
| --- | --- |
| version_id | Инкрементальный номер: v1.0, v1.1, v1.2... |
| effective_from | С какого периода применяется (следующий после утверждения) |
| effective_to | По какой период (NULL = текущая активная версия) |
| change_type | Объёмное / Стоимостное / Структурное |
| reason | Основание: номер версии РД, номер дополнительного соглашения |
| changed_by | Стройконтроль (инициатор) |
| approved_by | Директор проекта (утверждающий) |

## 12.2 Типы изменений и последствия

| **Тип** | **Что изменилось** | **Что пересчитывается** |
| --- | --- | --- |
| Объёмное | plan_volume по существующей позиции | planned_pace, forecast_weighted, forecast_critical, pct_ready |
| Стоимостное | contract_value по позиции (новый РДЦ) | weight_coef по всем позициям, ASSERT SUM == 1.0 |
| Структурное: новая позиция | Добавлена позиция в BoQ | История начинается с момента добавления. Нулевых исторических периодов нет. |
| Структурное: удаление позиции с нулевым фактом | Позиция без факта | Удаление разрешено. weight_coef пересчитываются. |
| Структурное: удаление позиции с ненулевым фактом | Позиция с историей факта | Удаление заблокировано. Варианты: перевести в статус «Исключена из scope» ИЛИ объединить (merge) через администратора + директора. |

## 12.3 Принцип неизменности истории

Закрытые периоды не пересчитываются — данные заморожены. Новая версия BoQ применяется начиная со следующего открытого периода.

Исключение — техническая ошибка при инициализации: ретроактивный пересчёт возможен только с решением директора. Все затронутые периоды маркируются «пересчитано [дата], основание».

## 12.4 Отображение изменений по ролям

| **Роль** | **Отображение** |
| --- | --- |
| Стройконтроль | В карточке вида работ: «1 200 м³ → 1 350 м³ [v2.0, дата]». История периодов с пометкой [база v1.0]. |
| Директор проекта | Баннер на дашборде при изменении базы. Вертикальная маркерная линия на графике % в точке изменения с подписью версии. |
| Администратор | Полный журнал версий BoQ со всеми атрибутами. |

# 13. Процедура обновления плановых объёмов (UpdateBaseline)

Изменение планового объёма вида работ в середине строительства выполняется по двухшаговой процедуре. Изменение применяется только между периодами — если текущий период открыт, утверждение заблокировано.

| **Шаг** | **Участник** | **Действие** |
| --- | --- | --- |
| 1. Инициация | Стройконтроль | Создаёт запрос: вид работ, новый плановый объём, основание (обязательно), документ-основание (версия РД или доп. соглашение, обязательно). Статус запроса: «Ожидает утверждения». Данные в систему не вносятся. |
| 2. Утверждение | Администратор | Проверяет запрос. Если период открыт — утверждение заблокировано («Дождитесь закрытия периода P[N]»). Принимает решение: утвердить или отклонить с комментарием. |
| 3a. Утверждено | Система | Применяет изменение: новый plan_volume, пересчёт weight_coef (ASSERT SUM==1.0), planned_pace, forecast. Маркировка на дашборде: «план скорректирован [дата]». Аудит-лог. Исторические периоды не пересчитываются. |
| 3b. Отклонено | Система | Данные не изменяются. Стройконтроль получает уведомление с комментарием администратора. |
