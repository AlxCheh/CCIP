# Auth & Security Architecture

## 1. Purpose

Модуль `Auth & Security` обеспечивает безопасный доступ к платформе CCIP.

Его задачи:

- аутентификация пользователей;
    
- авторизация действий;
    
- контроль доступа по ролям;
    
- защита публичных точек входа;
    
- управление токенами доступа.
    

Основной принцип:

> Любое действие должно выполняться только после проверки подлинности субъекта и его прав доступа.

---

## 2. Responsibilities

Модуль отвечает за:

1. login/logout пользователей;
    
2. выпуск access token;
    
3. выпуск refresh token;
    
4. валидацию JWT;
    
5. RBAC контроль;
    
6. защиту публичных токенизированных сценариев;
    
7. security policies на уровне API.
    

---

## 3. Main Entities

### User

Пользователь системы.

Содержит:

- идентификатор;
    
- роль;
    
- статус активности;
    
- организационную принадлежность.
    

---

### Access Token

JWT токен для доступа к API.

Содержит:

- user id;
    
- role;
    
- tenant id;
    
- срок действия.
    

---

### Refresh Token

Токен продления сессии.

Содержит:

- hash токена;
    
- user id;
    
- expiry;
    
- revoked status.
    

---

### GP Token

Одноразовый токен подачи данных генерального подрядчика.

Содержит:

- token value;
    
- expiration time;
    
- submission status.
    

---

## 4. Roles

В системе существуют роли:

### `admin`

Полный административный доступ.

---

### `director`

## 5. Authentication Workflow

### 5.1 Login

При входе:

1. валидируются учетные данные;
    
2. создается access token;
    
3. создается refresh token;
    
4. refresh token сохраняется в системе.
    

Результат:

> пользователь получает доступ к защищенному API

---

### 5.2 Access Validation

При каждом запросе:

1. извлекается JWT;
    
2. валидируется подпись;
    
3. валидируется срок действия;
    
4. проверяется активность пользователя.
    

---

### 5.3 Refresh

При обновлении сессии:

1. валидируется refresh token;
    
2. проверяется revoked status;
    
3. генерируется новый access token;
    
4. refresh token ротируется.
    

---

### 5.4 Logout

При выходе:

1. refresh token помечается revoked;
    
2. сессия становится недействительной.
    

---

## 6. Authorization Workflow

После аутентификации:

1. определяется роль пользователя;
    
2. проверяется допустимость действия;
    
3. разрешается или запрещается доступ.
    

---

### RBAC Rule

> Если роль не имеет разрешения на действие — запрос отклоняется.

---

## 7. GP Token Workflow

Для сценария подачи данных подрядчиком:

1. создается одноразовый токен;
    
2. токен отправляется подрядчику;
    
3. подрядчик отправляет данные;
    
4. токен становится недействительным.
    

---

### Ограничения:

- токен одноразовый;
    
- токен ограничен по времени;
    
- токен дает доступ только к разрешенному действию.
    

---

## 8. Security Constraints

Обязательные ограничения:

1. все защищенные endpoints требуют JWT;
    
2. refresh token должен храниться безопасно;
    
3. GP token должен быть одноразовым;
    
4. RBAC обязателен для всех внутренних действий;
    
5. публичный токен не должен давать расширенных прав.
    

---

## 9. Security Policies

### API Protection

- все приватные маршруты защищены;
    
- публичные маршруты ограничены по назначению.
    

---

### Token Security

- access token короткоживущий;
    
- refresh token ротационный;
    
- все токены проверяются на срок действия.
    

---

### Session Security

- revoked токены запрещены;
    
- повторное использование запрещено.
    

---

## 10. Dependencies

### Object Lifecycle
