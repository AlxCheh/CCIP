# Sync Engine Architecture

## 1. Purpose

Модуль `Sync Engine` обеспечивает надежную синхронизацию данных между мобильным приложением и сервером.

Его задачи:

- поддержка offline-first сценариев;
    
- очередь операций синхронизации;
    
- обнаружение конфликтов;
    
- разрешение конфликтов;
    
- гарантированная доставка данных.
    

Основной принцип:

> Все изменения, выполненные offline, должны быть синхронизированы без потери данных и без недетерминированных конфликтов.

---

## 2. Responsibilities

Модуль отвечает за:

1. накопление offline операций;
    
2. отправку операций на сервер;
    
3. проверку конфликтов версий;
    
4. возврат конфликтных состояний;
    
5. подтверждение применённых операций;
    
6. повторную синхронизацию при сбоях;
    
7. ручное разрешение конфликтов.
    

---

## 3. Main Entities

### Sync Queue

Очередь локальных операций, накопленных offline.

Содержит:

- тип операции;
    
- payload;
    
- статус;
    
- версия записи;
    
- время создания.
    

---

### Sync Operation

Единичная операция синхронизации.

Примеры:

- submit_fact
    
- update_fact
    
- upload_photo
    
- resolve_conflict
    

---

### Conflict Record

Запись конфликта между локальными и серверными данными.

Содержит:

- локальное состояние;
    
- серверное состояние;
    
- причина конфликта.
    

---

### Sync State

Состояние процесса синхронизации.

Содержит:

- idle
    
- syncing
    
- conflict
    
- failed
    

---

## 4. Offline-first Workflow

### 4.1 Local Change

При изменении данных offline:

1. изменение сохраняется локально;
    
2. создается sync operation;
    
3. операция добавляется в sync queue.
    

Результат:

> пользователь продолжает работу независимо от сети

---

### 4.2 Sync Trigger

При восстановлении сети:

1. запускается sync manager;
    
2. выбираются pending операции;
    
3. операции отправляются на сервер.
    

---

### 4.3 Server Validation

На сервере:

1. валидируется операция;
    
2. проверяется версия записи;
    
3. проверяется допустимость изменения;
    
4. операция применяется или отклоняется.
    

---

### 4.4 Sync Result

После обработки:

- если операция успешна → статус `applied`
    
- если конфликт → статус `conflict`
    
- если ошибка → статус `failed`
    

---

## 5. Conflict Detection

Конфликт возникает если:

1. версия локальной записи отличается от серверной;
    
2. серверное состояние изменилось после локального редактирования;
    
3. операция нарушает инварианты.
    

---

### Основное правило:

> Конфликт определяется только по версии данных, а не по времени изменения.

---

## 6. Conflict Resolution Workflow

При обнаружении конфликта:

1. сервер возвращает конфликтные данные;
    
2. локальная очередь помечает запись как `conflict`;
    
3. пользователь выбирает вариант разрешения;
    
4. создается `resolve_conflict` операция;
    
5. операция отправляется повторно.
    

---

### Разрешение возможно через:

- принять серверную версию;
    
- применить локальную версию;
    
- объединить вручную.
    

---

## 7. Sync Constraints

Обязательные ограничения:

1. операции должны применяться последовательно;
    
2. конфликт определяется только по версии;
    
3. last-write-wins запрещён;
    
4. операция должна быть идемпотентной;
    
5. потеря очереди недопустима.
    

---

## 8. Idempotency Rules

Каждая sync operation должна быть идемпотентной.

Это означает:

1. повторная отправка не должна дублировать эффект;
    
2. операция должна иметь уникальный идентификатор;
    
3. сервер должен распознавать повтор.
    

---

### Правило:

> Повторная синхронизация одной операции не должна изменять данные дважды.

---

## 9. Queue Processing Rules

Очередь синхронизации должна:

1. обрабатывать операции FIFO;
    
2. останавливать пакет при конфликте;
    
3. сохранять неуспешные операции;
    
4. продолжать после устранения ошибки.
    

---

### Ограничение:

> Нельзя применять следующие операции, если предыдущая конфликтует.

---

## 10. Photo Sync Workflow

Для медиафайлов:

1. фото сохраняется локально;
    
2. создается upload operation;
    
3. при сети файл загружается на сервер;
    
4. сервер подтверждает загрузку.
    

---

### Ограничения:

- файл должен иметь статус загрузки;
    
- повторная загрузка должна быть безопасной;
    
- метаданные синхронизируются вместе с файлом.
    

---

## 11. Failure Recovery

При сбоях:

1. queue сохраняется локально;
    
2. failed операции остаются в очереди;
    
3. sync manager выполняет повтор;
    
4. конфликтные операции требуют ручного вмешательства.
    

---

### Правило:

> Ошибка синхронизации не должна приводить к потере локальных данных.

---

## 12. Dependencies

Модуль зависит от:

### Mobile Architecture

Для локального хранения и очереди.

### Data Layer

Для проверки версий и применения операций.

### Period Engine

Для проверки допустимости изменений периода.

### Auth & Security

Для аутентификации sync операций.

## 14. Read Policy

При изменении логики синхронизации:

1. читать только нужный раздел;
    
2. подключать только связанный ADR;
    
3. не читать другие архитектурные модули без необходимости.
    

---

## 15. Main Principle

> Синхронизация должна быть идемпотентной, последовательной и устойчивой к сетевым сбоям без потери данных.
