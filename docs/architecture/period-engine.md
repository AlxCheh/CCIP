# Period Engine Architecture

## 1. Purpose

Модуль `Period Engine` управляет жизненным циклом отчетного периода в системе CCIP.

Его задачи:

- открытие периода;
    
- прием данных от участников;
    
- верификация фактов;
    
- закрытие периода;
    
- запуск расчетов аналитики.
    

Основной принцип:

> Период может переходить в следующее состояние только при соблюдении всех инвариантов текущего состояния.

---

## 2. Responsibilities

Модуль отвечает за:

1. создание нового отчетного периода;
    
2. управление статусами периода;
    
3. прием данных генерального подрядчика;
    
4. прием фактов стройконтроля;
    
5. проверку допустимости закрытия периода;
    
6. запуск snapshot-расчетов;
    
7. блокировку некорректных переходов.
    

---

## 3. Main Entities

### Period

Основная сущность отчетного периода.

Содержит:

- идентификатор периода;
    
- статус;
    
- даты открытия и закрытия;
    
- токен подачи ГП;
    
- SLA дедлайны.
    

---

### Period Facts

Фактические данные по видам работ.

Содержит:

- объем ГП;
    
- объем стройконтроля;
    
- статус сверки;
    
- версию записи.
    

---

### Snapshot

Снимок состояния объекта после закрытия периода.

Содержит:

- готовность объекта;
    
- расчетные показатели;
    
- прогнозные даты.
    

---

## 4. Period States

Период может находиться в одном из состояний:

### `open`

Период открыт для подачи данных.

---

### `gp_submitted`

Генподрядчик подал данные.

---

### `verification`

Стройконтроль проводит сверку.

---

### `closed`

Период закрыт.

---

### `force_closed`

Период закрыт автоматически по SLA.

---

## 5. Workflow

### 5.1 Open Period

При открытии периода:

1. проверяется наличие утвержденного zero-report;
    
2. проверяется отсутствие другого открытого периода;
    
3. создается новый period;
    
4. генерируется GP token;
    
5. назначаются SLA дедлайны.
    

Результат:

> создается период в статусе `open`

---

### 5.2 GP Submission

После подачи данных ГП:

1. валидируется токен;
    
2. сохраняются данные ГП;
    
3. фиксируется время подачи;
    
4. статус меняется на `gp_submitted`
    

---

### 5.3 Verification

Во время верификации:

1. стройконтроль вносит свои данные;
    
2. сравниваются значения;
    
3. выявляются расхождения;
    
4. фиксируются discrepancies;
    
5. статус меняется на `verification`
    

---

### 5.4 Close Period

Перед закрытием:

1. проверяется отсутствие открытых расхождений;
    
2. выполняются аналитические расчеты;
    
3. создается readiness snapshot;
    
4. период переводится в `closed`
    

---

## 6. Invariants

Инварианты периода:

1. может существовать только один открытый период на объект;
    
2. нельзя закрыть период при открытых расхождениях;
    
3. нельзя открыть период без zero-report;
    
4. snapshot обязателен при закрытии;
    
5. переход состояния допускается только по workflow.
    

---

## 7. Constraints

### Ограничения открытия:

- только один активный период;
    
- zero-report должен быть approved.
    

### Ограничения закрытия:

- все обязательные данные внесены;
    
- отсутствуют unresolved discrepancies.
    

### Ограничения аналитики:

- snapshot создается в транзакции закрытия периода.
    

---

## 8. Dependencies

Модуль зависит от:

### Object Lifecycle

Для проверки zero-report.

### Disputes & SLA

Для обработки расхождений.

### Analytics Engine

Для создания snapshots и прогнозов.

### Data Layer

Для транзакционной целостности.
