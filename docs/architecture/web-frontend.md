# Web Frontend Architecture

## 1. Purpose

Web-приложение для стройконтроля и управления объектом.

Целевые пользователи:
- `director` — мониторинг прогресса объекта
- `supervisor` — ведение периодов, верификация работ
- `admin` — управление объектами и пользователями

---

## 2. Stack

React + TypeScript + React Query + Zustand. Приложение: `apps/web/`.

---

## 3. Application Modules

### Dashboard Director

Главный экран директора:
- сводный прогресс объекта (% готовности по весовым коэффициентам)
- два прогноза: линейный и взвешенный с датой расчёта
- флаг разрыва план/факт при отклонении > порога
- переключение между периодами

### Period Cycle (Supervisor)

Полный UI-цикл периода:
1. Открытие периода (кнопка + дата)
2. Ввод фактических работ по позициям BoQ
3. Чеклист закрытия (§7.3 Концепции): обязательные поля перед закрытием
4. Верификация — карточка с фотодоказательствами и геометками
5. Закрытие периода (блокируется если не все верификации пройдены)

### Карточка верификации (§7.1 Концепции)

- Просмотр фотофиксации от мобильного агента
- Геометка + timestamp
- Кнопки: Подтвердить / Отклонить
- Групповое подтверждение для `delta=0` позиций

### GP Form

Форма подрядчика через GpToken:
- доступна без полного JWT (GpToken flow, ADR-009)
- ввод данных по позициям BoQ
- отправка → статус `submitted`

### Disputes Journal

- список расхождений с SLA-статусами
- история эскалаций
- polling или WebSocket для обновления SLA таймеров

### BoQ Versioning UI

- просмотр версий базовой линии
- история UpdateBaseline с датами
- diff между версиями

## 4. UX Principles (из Концепции §11.1)

- Нагрузка стройконтроля ≤ 10 часов/неделю
- Мобильно-адаптивный дизайн
- Блокировки на критических действиях (confirmation dialog)
- Оптимистичные обновления для верификации
- Отображение state machine периода: progress steps / status badge

---

## 5. API Integration

- REST API через `apps/api`
- React Query для кэширования и инвалидации
- Обработка состояний period state machine (OPEN/LOCKED/DISPUTED/CLOSED)
- Human-readable сообщения ошибок (не коды)

---

## 6. Testing

- Playwright E2E — обязателен для golden path каждого workflow
- CI: запуск на каждый PR к main
