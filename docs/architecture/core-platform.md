# Core Platform Architecture

## 1. Purpose

Модуль `Core Platform` описывает верхнеуровневую архитектуру платформы CCIP и определяет основные bounded contexts системы.

Его задачи:

- определить архитектурные границы модулей;
    
- определить основные доменные сущности;
    
- определить правила взаимодействия модулей;
    
- задать системные инварианты;
    
- обеспечить единый high-level контекст платформы.
    

Основной принцип:

> Каждый архитектурный модуль должен иметь четкую зону ответственности и взаимодействовать с другими модулями только через определенные контракты.

---

## 2. Platform Goal

CCIP — интеллектуальная платформа контроля строительства.

Платформа обеспечивает:

1. фиксацию факта выполнения работ;
    
2. контроль отчетных периодов;
    
3. обработку расхождений;
    
4. расчет аналитики;
    
5. мобильный сбор данных;
    
6. прозрачность состояния объекта строительства.
    

Основная бизнес-цель:

> предоставить достоверную картину прогресса строительства на основе подтвержденных данных.

## 5. Domain Interaction Model

Взаимодействие доменов происходит по цепочке:

1. `Object Lifecycle`  
    инициализирует объект;
    
2. `Period Engine`  
    открывает период;
    
3. `Sync Engine`  
    собирает фактические данные;
    
4. `Disputes & SLA`  
    обрабатывает расхождения;
    
5. `Analytics Engine`  
    формирует показатели;
    
6. `Data Layer`  
    гарантирует консистентность.
    

---

### Основное правило:

> Каждый модуль изменяет только собственный bounded context.

## 6. Responsibility Boundaries

### Auth & Security

Отвечает только за:

- аутентификацию;
    
- авторизацию;
    
- контроль доступа.
    

---

### Period Engine

Отвечает только за:

- статусы периода;
    
- переходы периода;
    
- закрытие периода.
    

---

### Disputes & SLA

Отвечает только за:

- disputes;
    
- SLA;
    
- escalation logic.
    

---

### Analytics Engine

Отвечает только за:

- расчеты;
    
- snapshots;
    
- прогнозы.
    

---

### Sync Engine

Отвечает только за:

- offline queue;
    
- sync operations;
    
- conflicts.
    

---

### Data Layer

Отвечает только за:

- транзакции;
    
- audit;
    
- versioning;
    
- consistency.
    

---

## 7. System Invariants

Платформа должна соблюдать глобальные инварианты:

1. один активный период на объект;
    
2. все изменения данных аудируются;
    
3. подтвержденные данные immutable;
    
4. аналитика строится только на закрытых периодах;
    
5. конфликт данных должен быть разрешен до закрытия периода;
    
6. доступ ограничивается ролями.
    

---

### Правило:

> Нарушение системного инварианта блокирует выполнение бизнес-операции.

---

## 8. Inter-Module Contracts

Модули взаимодействуют через контракты:

### Period Engine → Analytics Engine

Передача факта закрытия периода.

---

### Period Engine → Disputes & SLA

Передача открытых расхождений.

---

### Sync Engine → Data Layer

Передача синхронизируемых изменений.

---

### Auth & Security → All Modules

Передача контекста доступа.

---

### Основной принцип:

> Ни один модуль не должен напрямую управлять внутренним состоянием другого модуля.

---

## 9. Transaction Ownership

Каждый модуль владеет своей бизнес-логикой, но транзакционная целостность обеспечивается через `Data Layer`.

Правило:

1. модуль инициирует бизнес-операцию;
    
2. `Data Layer` обеспечивает транзакцию;
    
3. результат передается следующему модулю.
    

---

### Основной принцип:

> Бизнес-логика принадлежит модулю, консистентность принадлежит Data Layer.

---

## 10. Architectural Flow

Базовый поток работы системы:

1. создается объект;
    
2. открывается период;
    
3. собираются факты;
    
4. разрешаются disputes;
    
5. закрывается период;
    
6. рассчитывается аналитика;
    
7. обновляется snapshot.
    

---

Это основной жизненный цикл платформы.

---

## 11. Cross-cutting Rules

Общие правила для всех модулей:

1. все действия должны быть аудируемы;
    
2. все изменения должны быть транзакционны;
    
3. все состояния должны быть детерминированы;
    
4. все конфликты должны быть разрешимы;
    
5. все роли должны быть валидированы.
    

---

## 12. Dependencies Model

Зависимости модулей:

- `Auth & Security` → глобальная зависимость
    
- `Data Layer` → глобальная зависимость
    
- `Period Engine` → зависит от Object Lifecycle
    
- `Disputes & SLA` → зависит от Period Engine
    
- `Analytics Engine` → зависит от Period Engine
    
- `Sync Engine` → зависит от Auth & Data Layer
    

---

### Правило:

> Зависимости должны быть направлены только по утвержденной архитектурной схеме.

## 14. Read Policy

При архитектурном анализе:

1. сначала читать `core-platform.md`;
    
2. затем определять релевантный bounded context;
    
3. загружать только нужный модуль;
    
4. подключать ADR только при необходимости.
    

---

### Запрещено:

- читать все модули без определения контекста;
    
- изменять границы модулей без ADR;
    
- смешивать ответственность bounded contexts.
    

---

## 15. Main Principle

> Core Platform определяет границы ответственности модулей, а каждый модуль реализует только свой bounded context.
