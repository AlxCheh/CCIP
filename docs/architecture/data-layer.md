# Data Layer Architecture

## 1. Purpose

Модуль `Data Layer` определяет правила хранения, изменения и согласованности данных в платформе CCIP.

Его задачи:

- управление доступом к данным;
    
- обеспечение транзакционной целостности;
    
- контроль версионности;
    
- обеспечение неизменяемости данных;
    
- аудит изменений;
    
- изоляция данных арендаторов.
    

Основной принцип:

> Все изменения данных должны быть консистентными, трассируемыми и защищёнными от неконтролируемого изменения.

---

## 2. Responsibilities

Модуль отвечает за:

1. транзакционные границы;
    
2. optimistic locking;
    
3. immutable records;
    
4. audit trail;
    
5. tenant isolation;
    
6. versioning;
    
7. access constraints;
    
8. integrity rules.
    

---

## 3. Main Entities

### Business Data

Основные данные системы:

- объекты;
    
- периоды;
    
- факты;
    
- отчёты;
    
- аналитика.
    

---

### Audit Log

Журнал всех изменений данных.

Фиксирует:

- действие;
    
- старое значение;
    
- новое значение;
    
- пользователя;
    
- причину;
    
- время.
    

---

### Versioned Record

Запись, имеющая номер версии.

Используется для:

- optimistic locking;
    
- conflict detection;
    
- предотвращения потери данных.
    

---

### Tenant Context

Контекст организации, определяющий границы доступа к данным.

---

## 4. Transaction Boundaries

Любое изменение данных выполняется в транзакции.

Транзакция должна включать:

1. бизнес-изменение;
    
2. обновление версии;
    
3. запись в audit log;
    
4. фиксацию инвариантов.
    

---

### Правило:

> Данные считаются изменёнными только если успешно завершены все шаги транзакции.

---

## 5. Transaction Rules

При изменении записи:

1. начинается транзакция;
    
2. проверяется версия;
    
3. выполняется изменение;
    
4. обновляется версия;
    
5. записывается audit log;
    
6. транзакция коммитится.
    

---

### Ограничение:

> Частичное выполнение транзакции запрещено.

---

## 6. Optimistic Locking

Для конкурентных изменений используется version-based locking.

Каждая изменяемая запись содержит:

- `version`
    

При обновлении:

1. клиент передает текущую версию;
    
2. сервер сравнивает версии;
    
3. при несовпадении возникает конфликт.
    

---

### Правило:

> Конкурентный конфликт определяется по версии записи, а не по времени.

---

## 7. Immutability Rules

Некоторые данные после фиксации становятся неизменяемыми.

К immutable данным относятся:

- закрытые period facts;
    
- audit log записи;
    
- readiness snapshots.
    

---

### Ограничения:

1. прямое изменение immutable записей запрещено;
    
2. изменение допускается только через специальный административный процесс;
    
3. любое исключение фиксируется в audit log.
    

---

### Основной принцип:

> Подтвержденные данные нельзя изменять напрямую.

---

## 8. Audit Trail

Каждое изменение данных должно быть зафиксировано в audit log.

Фиксируется:

1. кто изменил;
    
2. что изменил;
    
3. когда изменил;
    
4. почему изменил;
    
5. старое и новое значение.
    

---

### Audit Requirements:

1. audit log append-only;
    
2. update/delete audit log запрещён;
    
3. audit запись создаётся в той же транзакции.
    

---

### Правило:

> Если audit запись не создана — изменение считается недействительным.

---

## 9. Versioning Rules

Версионность применяется для:

1. периодических данных;
    
2. BoQ;
    
3. offline sync данных;
    
4. snapshots.
    

---

Каждое изменение:

1. увеличивает версию;
    
2. сохраняет связь с предыдущей версией;
    
3. не уничтожает историю.
    

---

### Ограничение:

> История версий должна быть восстановима.

---

## 10. Tenant Isolation

Все данные принадлежат конкретному tenant.

Каждая бизнес-запись содержит:

- `tenant_id`
    

При доступе:

1. определяется tenant context;
    
2. применяется tenant filter;
    
3. доступ ограничивается текущим tenant.
    

---

### Правило:

> Пользователь не может видеть данные другого tenant.

---

## 11. Integrity Constraints

Data layer обеспечивает обязательные инварианты:

1. один активный период на объект;
    
2. обязательный snapshot при закрытии;
    
3. корректная сумма весов;
    
4. валидные статусы переходов;
    
5. уникальность активных сущностей.
    

---

### Правило:

> Нарушение инварианта приводит к откату транзакции.

---

## 12. Administrative Overrides

Для immutable данных допускается только контролируемая корректировка.

Процесс включает:

1. запуск специальной admin процедуры;
    
2. указание причины;
    
3. запись override в audit log;
    
4. каскадный пересчёт зависимых данных.
    

---

### Ограничение:

> Прямой update immutable данных запрещён даже для администратора.

---

## 13. Failure Recovery

При ошибке транзакции:

1. изменения откатываются;
    
2. версия не увеличивается;
    
3. audit log не создаётся;
    
4. клиент получает ошибку.
    

---

### Правило:

> Невалидная транзакция не должна оставлять частичных следов в системе.

---

## 14. Dependencies

Модуль зависит от:

### Auth & Security

Для идентификации пользователя и tenant context.

### Period Engine

Для фиксации периодических данных.

### Analytics Engine

Для хранения snapshots и расчетных данных.

### Sync Engine

Для проверки версий и разрешения конфликтов.

## 16. Read Policy

При изменении data layer логики:

1. читать только нужный раздел;
    
2. подключать только релевантный ADR;
    
3. не читать другие архитектурные модули без необходимости.
    

---

## 17. Main Principle

> Любое изменение данных должно быть атомарным, проверяемым, версионируемым и полностью аудируемым.
