# Analytics Engine Architecture

## 1. Purpose

Модуль `Analytics Engine` отвечает за расчет аналитических показателей объекта строительства.

Его задачи:

- расчет готовности;
    
- формирование snapshots;
    
- прогнозирование завершения;
    
- расчет темпов выполнения;
    
- подготовка данных для dashboard.
    

Основной принцип:

> Все аналитические показатели должны рассчитываться на основе зафиксированных и верифицированных данных периода.

---

## 2. Responsibilities

Модуль отвечает за:

1. расчет готовности работ;
    
2. расчет готовности объекта;
    
3. формирование snapshots;
    
4. расчет темпа выполнения;
    
5. прогноз завершения;
    
6. обновление аналитических представлений.
    

---

## 3. Main Entities

### Work Readiness

Процент готовности отдельного вида работ.

---

### Object Readiness

Сводная готовность объекта.

---

### Snapshot

Зафиксированное состояние объекта после закрытия периода.

---

### Forecast

Прогноз завершения работ.

---

### Materialized View

Подготовленное агрегированное представление для dashboard.

---

## 4. Analytics Workflow

### 4.1 Period Close Trigger

После закрытия периода:

1. запускается расчет аналитики;
    
2. формируется snapshot;
    
3. обновляется dashboard.
    

---

### 4.2 Readiness Calculation

Для каждого вида работ:

1. определяется фактический объем;
    
2. определяется плановый объем;
    
3. рассчитывается готовность работы.
    

---

Формула:

> `work readiness = min(fact / plan, 100%)`

---

### 4.3 Object Readiness Calculation

Для объекта:

1. берутся readiness по работам;
    
2. применяется вес каждого вида работ;
    
3. рассчитывается суммарная готовность.
    

---

Формула:

> `object readiness = Σ(work readiness × weight)`

---

### 4.4 Pace Calculation

Для расчета темпа:

1. анализируются исторические значения;
    
2. рассчитывается темп выполнения;
    
3. учитывается окно периодов.
    

---

### 4.5 Forecast Calculation

Для прогноза:

1. определяется текущий темп;
    
2. рассчитывается остаток работ;
    
3. вычисляется прогнозная дата завершения.
    

---

## 5. Snapshots

Snapshot создается при закрытии периода.

Содержит:

- готовность работ;
    
- готовность объекта;
    
- прогнозы;
    
- темпы.
    

---

### Правило:

> Snapshot создается транзакционно вместе с закрытием периода.

---

## 6. Constraints

Обязательные ограничения:

1. аналитика строится только на закрытых периодах;
    
2. snapshot обязателен при закрытии;
    
3. live расчет dashboard запрещен;
    
4. прогноз строится только по валидным данным;
    
5. пересчет выполняется детерминированно.
    

---

## 7. Forecasting Rules

Прогноз строится на основе:

1. текущей готовности;
    
2. исторического темпа;
    
3. остаточного объема работ.
    

---

### Ограничения прогноза:

- отрицательный темп запрещен;
    
- нулевой темп требует специальной обработки;
    
- прогноз должен быть воспроизводим.
    

---

## 8. Dashboard Data Flow

Для dashboard:

1. analytics engine обновляет агрегаты;
    
2. materialized view обновляется;
    
3. dashboard читает только агрегаты.
    

---

### Основное правило:

> Dashboard не должен выполнять тяжелые аналитические расчеты в runtime.

---

## 9. Dependencies

Модуль зависит от:

### Period Engine

Источник закрытых периодов.

### Data Layer

Источник данных фактов и планов.

### Infrastructure

Для обновления materialized views и фоновых задач.

---

## 10. Related ADR

Связанные архитектурные решения:

- ADR по snapshots
    
- ADR по analytics consistency
    
- ADR по materialized views
    
- ADR по forecasting rules
    

---

## 11. Read Policy

При изменении аналитики:

1. читать только нужный раздел;
    
2. подключать только релевантный ADR;
    
3. не читать другие архитектурные модули без необходимости.
    

---

## 12. Main Principle

> Все аналитические показатели должны вычисляться детерминированно, транзакционно и только на подтвержденных данных.
