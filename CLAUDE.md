## НАЗНАЧЕНИЕ ПРОЕКТА

**CCIP (Construction Control & Intelligence Platform)** — SaaS-платформа для оцифровки и аналитики объектов капитального строительства.

**Ключевая ценность:** Заказчик видит верифицированный факт выполнения работ, а не данные ГП. Единственный оператор системы — строительный контроль Заказчика.

**Объект-пример:** Складской логистический центр класса А, стадия строительства.

## КАРТА ПРОЕКТА

### Четырёхуровневая модель данных

| Уровень | Название | Таблицы в БД |
|---------|----------|--------------|
| L0 | Конфигурация системы | `system_config` |
| L1 | Паспорт объекта | `objects`, `object_participants` |
| L2 | Проектная документация | `l2_documents`, `boq_versions`, `boq_items` |
| L3 | Текущий статус | `zero_reports`, `zero_report_items`, `periods`, `period_facts`, `discrepancies`, `photos` |

### Алгоритмические блоки и их зоны ответственности

| Блок | Функция | Статус |
|------|---------|--------|
| A — Инициализация | L0–L2: конфиг, паспорт, BoQ, версионирование | Схема готова |
| B — 0-отчёт | Стартовый факт, перекрёстная верификация, таймер | Схема готова |
| C — Цикл периода | Открытие → ГП → верификация SC → закрытие | Схема готова |
| D — Расхождения и SLA | Тип 1/2/3, deadlock A/B | Схема готова (`sla_events`) |
| E — Аналитика | % готовности, WMA темп, два прогноза, флаги | Схема готова (`readiness_snapshots`, `work_pace`) |
| F — UpdateBaseline | Двухшаговое изменение плановых объёмов | Схема готова (`baseline_update_requests`) |
| G — Версионирование L2 | Любое изменение BoQ → новая версия | Схема готова (`boq_versions`) |
| H — Смена ГП | SCD Type 2 в `object_participants` | Схема готова |
| I — Офлайн-синхронизация | Локальная очередь → конфликт → решение SC | Схема готова (`sync_queue`) |

## КЛЮЧЕВЫЕ АРХИТЕКТУРНЫЕ РЕШЕНИЯ

### Неизменность истории
Закрытые периоды (`periods.status = 'closed'`) **не обновляются** физически.
Исправление возможно только через `audit_log` + маркировку `'пересчитано [дата], основание'`.
Исключение: техническая ошибка инициализации — только по решению директора.

### Весовые коэффициенты
`ASSERT SUM(weight_coef) == 1.0` — обеспечивается триггером `trg_boq_items_weight_coef`.
Веса привязаны к `contract_value`, **не пересчитываются при перевыполнении**.
Источник: РДЦ (при отсутствии — ССР временно).

### SLA-таймеры deadlock
Реализованы через `sla_events`. Планировщик (APScheduler) сканирует по индексу `idx_sla_events_scheduled`.

| Сценарий | День 0 | День 3 | День 5 / 14 |
|----------|--------|--------|-------------|
| A (молчание ГП) | Тип 2 выставлен | Уведомление директору | День 5: принудительное закрытие SC |
| B (активный спор) | ГП ответил, SC не принял | Уведомление директору | День 14: объём SC как консервативная оценка |

### % готовности
```
pct_ready(work) = MIN(fact_volume / plan_volume × 100, 100)
obj_readiness   = Σ(pct_ready(work) × weight_coef)
```
Никогда не превышает 100% на уровне вида работ. Хранится в `readiness_snapshots`.

### Взвешенный темп (WMA)
```
pace_weighted = Σ(volume[i] × decay_factor^i) / Σ(decay_factor^i)
```
- Плановые паузы (тип А) — исключаются (`work_pace.is_excluded = TRUE`).
- Внеплановый простой (тип Б) — включается, уведомление директору.
- Позиция не начата (тип В) — исключается.
- Таблица: `work_pace`.

### Два прогноза
| Прогноз | Формула | Колонка |
|---------|---------|---------|
| Взвешенный | `weighted_forecast(all_works)` | `readiness_snapshots.weighted_forecast_date` |
| Критический путь | `MAX(forecast_end)` по `weight_coef ≥ weight_threshold` ∪ `is_critical` | `readiness_snapshots.critical_path_forecast_date` |

Флаг разрыва: `gap_flag = TRUE` когда разрыв ≥ `forecast_gap_alert` периодов.

### Дашборд директора
Читается из материализованного представления `mv_object_current_status`.
Обновляется `REFRESH MATERIALIZED VIEW CONCURRENTLY` после закрытия каждого периода.

### Офлайн-синхронизация
- Все действия SC на устройстве пишутся в `sync_queue` с `client_timestamp`.
- Закрытие периода — **только онлайн**.
- При конфликте — `sync_queue.status = 'conflict'`, решение за SC (`resolved_by`).
- `last-write-wins` не применяется.

### Версионирование BoQ
- Любое изменение L2 → новая запись в `boq_versions`.
- Версии нумеруются инкрементально (`version_number`: `1.0`, `1.1` ...).
- `boq_items.work_lineage_id` (UUID) — стабильный идентификатор позиции через смены версий.
- Только одна активная версия: `uq_boq_versions_active` (partial unique index).

### Аудит-лог
- Таблица `audit_log` — append-only, секционирована по месяцам (`PARTITION BY RANGE`).
- Дефолтная партиция `audit_log_default` перехватывает строки вне явных партиций.
- Ежемесячные партиции создаёт DBA или `pg_partman`.

| `sync_queue` | Офлайн-очередь действий мобильного приложения |
| `mv_object_current_status` | Материализованное представление для дашборда директора |

## ПАРАМЕТРЫ СИСТЕМЫ (L0, дефолты)

| Параметр | Дефолт | Назначение |
|----------|--------|-----------|
| `N_flag_threshold` | 3 | Мин. Тип-2 расхождений для флага Тип 3 |

| `overrun_warning_limit` | 20% | Жёсткий стоп при перевыполнении |

| `spike_threshold` | 2.0 | Порог выброса темпа (×WMA) |
| `zero_report_alert_days` | 5 | Дней до уведомления об утверждении 0-отчёта |

## ГЛОССАРИЙ

| Термин | Определение |
|--------|-------------|
| ОКС | Объект капитального строительства |
| ГП | Генеральный подрядчик |
| SC | Строительный контроль (stroycontrol) |
| BoQ | Ведомость объёмов работ (Bill of Quantities) |
| РДЦ | Расчёт договорных цен (источник contract_value) |
| ССР | Сводный сметный расчёт (резервный источник при отсутствии РДЦ) |
| 0-отчёт | Стартовый факт на дату подключения системы |
| Период | Единица времени для обновления данных (рекомендуется 1 неделя) |
| Плановая пауза | Период без работ (тип А), исключается из WMA |
| Верифицированный факт | Объём, подтверждённый SC. Единственное основание для ввода в систему |
| Расхождение Тип 1 | Техническое: работы видны, замер возможен без разрушения |
| Расхождение Тип 2 | Спорное: зона скрыта / недоступна / требует разрушения |
| Расхождение Тип 3 | Системное: ≥N расхождений Тип 2 за M периодов (флаг на дашборде) |
| UpdateBaseline | Процедура изменения plan_volume: инициирует SC, утверждает Admin |
| work_lineage_id | UUID — стабильный идентификатор позиции BoQ через смены версий |
| decay_factor | Коэффициент затухания WMA (дефолт 0.9) |
| Критический путь | MAX(forecast_end) по видам работ с weight_coef ≥ weight_threshold или is_critical |
